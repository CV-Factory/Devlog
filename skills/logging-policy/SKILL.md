## Rules

- **정보성 로그 최소화**: 단순 `console.log` 또는 `Log.info`를 남발하지 않으며, 작업의 성공 여부나 진행 상태는 사용자 UI/CLI를 통해 전달한다.
- **에러 컨텍스트 포함**: 오류 발생 시 `Log.error` 또는 catch 블록 내에서 오류 원인 분석에 필요한 컨텍스트(입력값, 상태 등)를 함께 기록한다.
- **구조화된 로깅**: `Log.create({ tag: value })` 또는 `Log.provide`를 사용하여 로그에 맥락(context)을 부여한다.
- **보안 준수**: 비밀번호, API 키, 개인정보 등 민감 데이터는 로그에 포함하지 않는다.
- **대화 원문 로그 금지**: Devlog 수집/가공 과정에서 대화 원문 전체를 운영 로그로 남기지 않는다.
- **가공/저장 전 정리**: 배포 대상(`public`) 컨텐츠는 레드랙션/마스킹 정책을 적용한 후 저장한다.

## Checklist

- **로그가 필요한 상황인가?**: 운영 중 에러 분석이 필요한 경우에만 로그를 남기는가?
- **충분한 컨텍스트가 있는가?**: 에러 발생 시점의 파라미터나 상태 정보가 포함되었는가? (민감 정보 제외)
- **Log 유틸리티 사용**: 프로젝트 내 공통 `Log` 유틸리티(존재하는 경우)를 활용하는가?
- **로그 레벨 준수**: `info`는 최소화하고, 실제 문제 상황은 `error` 레벨로 기록하는가?
- **민감 내용 보호**: Devlog 수집 payload나 렌더 결과에서 민감정보가 제거되었는가?
